# Profiling Finding

Recorded the Profiler while opening a room detail page with a 5-item wishlist and typing into the wishlist item name inputs. The most notable finding: every keystroke in any item's "Item Name" input caused the entire `WishlistEditor` to commit a re-render — including all sibling item rows that weren't being edited. The root cause is that `react-hook-form`'s `useFieldArray` fields array is a stable reference but the parent `WishlistEditor` component re-renders on each `onChange` event because `formState.errors` is recalculated (even when no new errors appear). Each commit was fast (~4 ms) given the small list, but at 50+ items the `.map()` over all field rows would accumulate render cost linearly. Wrapping each item row in `React.memo` (passing only its own `field` and error slice as props) or replacing the `onChange` re-render with `mode: 'onBlur'` (already set) would limit the blast radius to the single changed row.
